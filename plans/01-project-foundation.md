## Step 1: Initialize the Expo Project

> **Important:** `@react-native-firebase` requires native modules that are not available in Expo Go. This project uses **Continuous Native Generation (CNG)** — `ios/` and `android/` directories are generated by `expo prebuild` and should not be manually edited. Do NOT use the `bare-minimum` template; use the default template with CNG.

1. Run `npx create-expo-app@latest triptrack` (default template — TypeScript, expo-router pre-configured)
2. Update `tsconfig.json` to add the path aliases used throughout the project:
   ```json
   {
     "extends": "expo/tsconfig.base",
     "compilerOptions": {
       "strict": true,
       "paths": {
         "@/*": ["./src/*"],
         "@/tw": ["./src/tw/index"],
         "@/tw/image": ["./src/tw/image"],
         "@/tw/animated": ["./src/tw/animated"]
       }
     }
   }
   ```
3. Generate the native projects:
   ```bash
   npx expo prebuild
   ```
   This creates `ios/` and `android/` from the config plugins. Re-run after adding any new native package.
4. Verify the project builds:
   ```bash
   npx expo run:ios   # requires Xcode
   npx expo run:android  # requires Android Studio
   ```
   **Do NOT use `npx expo start` with Expo Go** — `@react-native-firebase` and ML Kit require a custom build.

## Step 3: Configure NativeWind v5 / Tailwind CSS v4

NativeWind v5 uses **Tailwind CSS v4** and the **react-native-css** runtime. The key differences from v4:
- No `tailwind.config.ts` — theme tokens are defined in CSS `@theme` blocks inside `global.css`
- No babel preset for NativeWind — Metro handles it via `withNativewind`
- All React Native core components (`View`, `Text`, etc.) must be wrapped with `useCssElement` from `react-native-css` to accept `className`. These wrappers live in `src/tw/` (Step 6.5)
- `babel.config.js` is not needed if it only contains the Expo preset — delete it

### 1. `metro.config.js`

> **NativeWind v5 change:** The `input` option has been removed from `withNativewind` in v5. CSS-first configuration is now handled entirely in `global.css` via `@import` directives. Just call `withNativewind(config)` with no second argument.

```js
const { getDefaultConfig } = require("expo/metro-config");
const { withNativewind } = require("nativewind/metro");

const config = getDefaultConfig(__dirname);

module.exports = withNativewind(config);
```

### 2. `postcss.config.mjs`

```js
export default {
  plugins: {
    "@tailwindcss/postcss": {},
  },
};
```

### 3. `global.css`

```css
@import "tailwindcss/theme.css" layer(theme);
@import "tailwindcss/utilities.css" layer(utilities);

/* All Material Design 3 tokens and Lexend font are defined here — see Step 4 */
```

### 4. Delete `babel.config.js` if it only contains the default Expo preset

If your `babel.config.js` only has:
```js
module.exports = { presets: ['babel-preset-expo'] };
```
Delete the file entirely. Expo SDK 53+ with Metro handles this without an explicit babel config. If you have other transforms (e.g., `react-native-reanimated/plugin`), keep only those.

### 5. Root layout import

In `app/_layout.tsx`, import `global.css` and the `expo-sqlite` localStorage polyfill at the top:
```typescript
import "../global.css";
import "expo-sqlite/localStorage/install"; // global localStorage backed by SQLite (used by driveService queue — Plan 10)
```

## Step 6.5: Create `src/tw/` CSS-Wrapped Component Exports

> **Critical:** NativeWind v5 requires all React Native core components to be wrapped with `useCssElement` from `react-native-css` before they can accept a `className` prop. Raw `react-native` components will silently ignore `className`. All code in this project **must** import `View`, `Text`, `ScrollView`, `Pressable`, `TextInput`, and `Image` from `@/tw` (not from `react-native` or `react-native-reanimated`).

## Step 7: Define TypeScript Types

> **Two-layer type system:** This project has two type layers that serve different purposes:
> - `src/types/index.ts` (this file) — **UI view models**. Lightweight interfaces shaped for component props, store state, and local computations. May omit Firestore-specific fields like `syncStatus` or `_pendingWrite`.
> - `src/types/firestore.ts` — **Firestore document shapes** (defined in Plan 09). These are the canonical data structures with all `userId`, `tripId`, `syncStatus`, timestamps, and server-side fields. Use Firestore types whenever reading/writing to Firebase.
>
> Services and stores bridge between the two layers: services accept/return Firestore types, stores expose UI types derived from Firestore types.

Create `src/types/index.ts`:

```typescript
export type ReceiptCategory = "food" | "travel" | "warranty" | "utility" | "shopping" | "other";

export interface ReceiptItem {
  name: string;
  quantity: number;
  price: number;
}

// UI view model — see FirestoreReceipt in src/types/firestore.ts for the Firestore shape
export interface Receipt {
  id: string;
  merchant: string;
  date: string;
  amount: number;
  category: ReceiptCategory;
  items?: ReceiptItem[];
  imageUri?: string;
  isWarranty: boolean;
  syncStatus?: "synced" | "pending" | "error";
}

// UI view model — see FirestoreWarranty in src/types/firestore.ts
export interface Warranty {
  id: string;
  receiptId: string;
  productName: string;
  manufacturer: string;
  purchaseDate: string;
  expirationDate: string;
  coverageType: string;
}

export interface TripItem {
  id: string;
  name: string;
  description: string;
  category: string;
  assignedTo: string | null;
  status: "unassigned" | "assigned" | "brought";
}

export interface TripCategory {
  id: string;
  name: string;
  icon: string;
  variant: "default" | "tertiary" | "full-width";
  items: TripItem[];
}

// Participant model — matches TripParticipant from Plan 07
// Supports both AppUser (has uid) and GhostParticipant (uid undefined, managed by another user)
export interface TripParticipant {
  id: string;           // participantId (client-generated UUID)
  uid?: string;         // Firebase Auth UID if an AppUser
  name: string;
  email?: string;
  avatarUri?: string;
  isGhost: boolean;     // true = GhostParticipant (no Firebase account)
  managedBy?: string;   // uid of AppUser who manages this ghost
  amountPaid: number;
  amountOwed: number;
}

// Keep Participant as alias for backward compat; prefer TripParticipant in new code
export type Participant = TripParticipant;

// UI view model — see FirestoreTrip in src/types/firestore.ts
export interface Trip {
  id: string;
  name: string;
  startDate: string;
  endDate: string;
  participants: TripParticipant[];
  totalSpend: number;
  categories: TripCategory[];
  carpools: Carpool[];
  settlements: SettlementTransaction[];
  totalPot: number;
  categoryBreakdown: Record<string, number>;
}

// UI view model — see FirestoreExpense in src/types/firestore.ts
// description field is required here and in FirestoreExpense (added in Plan 09)
export interface Expense {
  id: string;
  tripId?: string;
  receiptId?: string;
  description: string;        // human-readable label, e.g. "Dinner at The Lobster Shack"
  amount: number;
  paidBy: string;             // participantId
  splitAmong: string[];       // participantIds
  splitType: "equal" | "custom" | "percentage";
  customAmounts?: Record<string, number>;
}

export interface Carpool {
  id: string;
  tripId: string;
  name: string;
  route: string;
  distance: number;
  fuelCost: number;
  passengers: CarpoolPassenger[];
}

export interface CarpoolPassenger {
  participantId: string;
  role: "driver" | "navigator" | "passenger";
  amountOwed: number;
  settled: boolean;
}

export interface SettlementTransaction {
  id: string;
  tripId: string;
  fromParticipantId: string;
  toParticipantId: string;
  amount: number;
  status: "pending" | "settled";
}
```

```typescript
import auth from "@react-native-firebase/auth";

/**
 * Returns the current user's UID or throws if unauthenticated.
 * Use this instead of auth().currentUser!.uid to get a clear error.
 */
export function requireAuth(): string {
  const uid = auth().currentUser?.uid;
  if (!uid) throw new Error("requireAuth: no authenticated user");
  return uid;
}
```

```javascript
};
```

### Zustand store mocking pattern

```typescript
// In test files, reset stores between tests:
import { useReceiptStore } from "@/stores/receiptStore";

beforeEach(() => {
  useReceiptStore.setState({ receipts: [], loading: false });
});
```

### Test file conventions

- Test files: `src/**/__tests__/*.test.ts` or `*.test.tsx` co-located with the module
- Service tests: mock Firestore and verify calls with `expect(mockFirestore.collection).toHaveBeenCalledWith("receipts")`
- Component tests: use `render` + `fireEvent` from `@testing-library/react-native`
- **TDD workflow:** write a failing test first, then implement to make it pass
