## 2. Technology Choices

### 2.1 Firebase (core backend)

Firebase is the single backend platform.

| Requirement | Firebase Solution |
|---|---|
| Real-time collaboration on trips | Firestore real-time listeners |
| Offline use during trips (poor signal) | Firestore offline persistence (built-in) |
| Receipt image sharing within trips | Firebase Storage with security rules |
| Google Sign-In for auth AND Drive API | Firebase Auth + Google OAuth (single token flow) |
| Apple Sign-In (App Store requirement) | Firebase Auth with Apple provider |
| Push notifications for settlement reminders | Firebase Cloud Messaging (FCM) |
| Generous free tier for small-group apps | Spark plan: 50K reads/day, 1GB Firestore, 5GB Storage |

> **CNG note:** This project uses **Continuous Native Generation (CNG)** — `ios/` and `android/` directories are generated by `npx expo prebuild` from `app.config.ts`. Never use `bare-minimum` template or manually edit the native directories. `@react-native-firebase` requires this native setup; Expo Go will not work.

> **API note:** This project uses `@react-native-firebase` (not the Firebase JS Web SDK). All Firestore calls use `firestore()` factory functions — **not** `getFirestore()` / `collection()` from `"firebase/firestore"`.

### 2.2 Google ML Kit (on-device OCR)

ML Kit's Text Recognition runs entirely on-device — no network required. This is critical because receipts are often scanned somewhere with poor connectivity.

Package: `@infinitered/react-native-mlkit-text-recognition` (v5.0.1+ for SDK 54 — the official Expo-compatible CNG wrapper).

### 2.3 Google Drive API (personal receipt backup)

Google Drive is a **personal, user-owned backup** for receipt images — not primary storage. Firebase Storage is primary; Drive is a secondary copy the user controls.

- Requires `https://www.googleapis.com/auth/drive.file` OAuth scope, added to the Google Sign-In flow
- Accessed via Drive REST API (`https://www.googleapis.com/upload/drive/v3/files`)
- No additional package needed — `@react-native-google-signin/google-signin` provides the access token

## 3. Hybrid Participant Model

Trips can include two types of participants:

### AppUser
- Has a Firebase Auth account (`uid`)
- Installed the app and joined/created the trip
- Sees the trip in real-time, enters expenses, claims planner items, settles debts
- Receives push notifications for reminders and settlement requests
- Owns their own receipts and can back them up to Google Drive

### GhostParticipant
- Just a name and optional contact info (email, phone)
- Created by an AppUser when adding someone without the app
- Expenses and settlements are managed on their behalf by any AppUser in the trip
- Cannot log in or see the app
- **Upgrade path:** if a GhostParticipant later installs the app, they join via a trip invite link. A Cloud Function detects their email matches a ghost entry and merges the records (see Plan 04 Step 9).

## 4. Service Layer Design

The service layer sits between Zustand stores and Firebase. Stores hold reactive UI state; services handle all I/O. This separation means:

- Stores can be unit-tested without Firebase
- Services can be swapped without touching UI code
- Offline behavior is handled once, not scattered across screens

### Data flow

```
UI Component → Zustand store (read)
UI Component → Service (write/action)
Firestore listener → syncService → Zustand store (update)
```

### File structure

```
src/
  services/
    firebase.ts          # Firebase app initialization, export db/auth/storage
    utils.ts             # requireAuth() — already implemented in Plan 01
    authService.ts       # Sign-in, sign-out, session, token management
    userService.ts       # User profile CRUD in Firestore
    receiptService.ts    # Receipt CRUD, local save, Storage upload
    warrantyService.ts   # Warranty CRUD, notification scheduling
    tripService.ts       # Trip CRUD, participant management, invitations + expense CRUD
    settlementService.ts # Settlement optimization algorithm
    driveService.ts      # Google Drive backup queue management
    ocrService.ts        # ML Kit on-device OCR
    syncService.ts       # Firestore real-time listener lifecycle
```

> **No `expenseService.ts`:** Expense functions live in `tripService.ts` — expenses are always subcollections of a trip document.

### Core pattern: service writes, listener updates Zustand

```typescript
import firestore from "@react-native-firebase/firestore";

// Services write to Firestore:
await tripService.addExpense(tripId, expenseData);

// syncService sets up a listener (called once at trip open):
const unsubscribe = firestore()
  .collection("trips").doc(tripId).collection("expenses")
  .onSnapshot(snapshot => {
    const expenses = snapshot.docs.map(d => ({ id: d.id, ...d.data() }));
    useTripStore.getState().setExpenses(tripId, expenses);
  });

// UI reads from Zustand (no direct Firebase calls in components):
const expenses = useTripStore(state => state.expenses[tripId]);
```

## 5. Offline-First Strategy

### Firestore offline persistence

Enabled by default in `@react-native-firebase`. All reads serve from cache when offline; all writes are queued and applied when connectivity returns. No extra configuration required beyond optionally setting unlimited cache size:

```typescript
// src/services/firebase.ts
import firestore from "@react-native-firebase/firestore";
firestore().settings({ persistence: true, cacheSizeBytes: firestore.CACHE_SIZE_UNLIMITED });
```

### What works offline

| Feature | Offline behavior |
|---|---|
| View trip expenses | Reads from Firestore cache |
| Add/edit an expense | Queued locally, synced on reconnect |
| View planner items | Reads from Firestore cache |
| Claim a planner item | Optimistic update, queued for sync |
| Scan and save a receipt | Saves locally; Firestore + Storage upload queued |
| View warranties | Reads from Firestore cache |

### What requires connectivity

| Feature | Why |
|---|---|
| Trip invitation acceptance | Must verify invite code against server |
| Push notification delivery | Requires FCM connection |
| Google Drive backup | REST API requires network |
| First-time sign-in | Firebase Auth token exchange |

### Upload queue for images

Receipt images (Firebase Storage + Google Drive) are queued in `expo-sqlite localStorage` when offline and processed in the background by `driveService.ts` and `receiptService.ts` on reconnect. See Plan 06 for the full queue implementation.

> **Important:** The upload queue uses `expo-sqlite localStorage` (the polyfill installed in `app/_layout.tsx` as `import 'expo-sqlite/localStorage/install'`). Never use `@react-native-async-storage/async-storage` — it is deprecated in this project.

## 6. Environment Setup

### 6.1 Firebase project

1. Create a new Firebase project at [console.firebase.google.com](https://console.firebase.google.com)
2. Enable **Authentication** — add Google and Apple providers
3. Enable **Firestore Database** — start in production mode (security rules in Plan 05)
4. Enable **Storage** — default bucket
5. Enable **Cloud Messaging** — for push notifications
6. Download `google-services.json` → place in `android/app/`
7. Download `GoogleService-Info.plist` → place in `ios/`

These files are already listed in `.gitignore` (from Plan 01 Step 11).

### 6.2 Google Cloud Console (for Drive API)

1. Go to [console.cloud.google.com](https://console.cloud.google.com) (same project as Firebase)
2. Enable the **Google Drive API**
3. Add OAuth scope `https://www.googleapis.com/auth/drive.file` to the consent screen
4. Add the SHA-1 certificate fingerprint for Android in the OAuth 2.0 client

### 6.3 Environment variables

Already covered in Plan 01. Required in `.env`:

```
EXPO_PUBLIC_GOOGLE_WEB_CLIENT_ID=...   # from Google Cloud Console OAuth client
EXPO_PUBLIC_FIREBASE_PROJECT_ID=...
```

## 8. Firebase Cloud Functions Setup

The `resolveGhostParticipant` function (Plan 04 Step 9) requires the Firebase Cloud Functions runtime. Set it up now.

```bash
# From the project root
firebase init functions
# Select: TypeScript, ESLint: yes, install dependencies: yes
```

This creates a `functions/` directory with its own `package.json`, `tsconfig.json`, and `src/index.ts`.

### Install server-side dependencies

```bash
cd functions
npm install firebase-functions@^6 firebase-admin@^13
npm install --save-dev @types/node
```

### `functions/src/index.ts`

```typescript
import * as admin from "firebase-admin";
admin.initializeApp();

export { resolveGhostParticipant } from "./resolveGhostParticipant";
```

### `firebase.json`

Ensure functions are configured with Node 20 runtime:

```json
{
  "functions": {
    "source": "functions",
    "runtime": "nodejs20"
  }
}
```
